# DispatchTalon 30-Day Pilot Proof Plan

Status: Internal operating plan
Use: Pilot activation, evidence capture, advisor follow-up, paid-pilot decision support

## Goal

Convert active pilot access into commercial evidence.

The goal is not to build more features. The goal is to prove whether DispatchTalon helps real operators structure dispatch decisions, setup data, site logs, worker credentials, and reviewable allocation reasoning.

## Week 1: Controlled Activation

Objective: Get the right users into a clean, understandable workflow.

Actions:

- send access to selected pilot users and advisors
- confirm login and password change
- guide users through dashboard, workers, jobs, Daily Site Log, Setup Uploads, and Our Business
- explain the core wedge: suitability before availability
- record first impressions within 48 hours
- prevent weak setup from becoming product judgement

Evidence to capture:

- access sent date
- login status
- first workflow opened
- first confusion point
- first "this could help" moment

## Week 2: Workflow Validation

Objective: Test practical use, not opinions alone.

Actions:

- ask pilot users to create or review one site log
- ask users to sign workers in and out in demo or safe pilot context
- ask users to add or review a custom credential type
- ask users to print a Daily Site Log report
- ask users to review source upload as onboarding relief
- ask advisors to challenge the category and claim boundaries

Evidence to capture:

- site logs created
- sign in/out use
- printed reports
- custom credential types added
- source upload interest
- friction notes

## Week 3: Commercial Proof

Objective: Identify whether the product solves a business problem strong enough for paid continuation.

Actions:

- ask what workflow they would use weekly
- ask what records they still manage in memory or spreadsheets
- ask what would make the system trusted
- ask whether another decision-maker should see it
- ask what should not be built yet
- document objections without defending the product too early

Evidence to capture:

- serious workflow problem identified
- request for adaptation
- willingness to involve another decision-maker
- referral possibility
- paid pilot interest
- strongest objection

## Week 4: Conversion And Decision

Objective: Decide continue, convert, narrow, or stop.

Actions:

- review usage and feedback trackers
- separate "nice feature" from "commercial problem"
- decide which pilots deserve support time
- prepare paid pilot offer only where evidence supports it
- stop or park weak-fit leads
- update claim-safe proof language

Evidence to capture:

- repeat login
- workflow completion
- follow-up meeting requested
- decision-maker involvement
- paid pilot likelihood
- quote permission
- product risk requiring fix

## Metrics To Track

- logins
- source uploads
- site logs created
- workers signed in/out
- daily reports printed
- custom credential types added
- personalisation forms completed
- serious feedback items
- referral offers
- paid pilot interest

## Feedback Categories

- workflow clarity
- onboarding relief
- worker suitability
- asset visibility
- credential visibility
- Daily Site Log value
- SmartRank trust
- print/report usefulness
- buyer trust
- technical risk
- commercial objection
- do-not-build warning

## Follow-Up Cadence

Use a controlled cadence:

- Day 0: access sent
- Day 1-2: login and first-impression check
- Day 5-7: workflow validation check
- Day 14: commercial problem check
- Day 21: decision-maker/referral check
- Day 30: continue, convert, narrow, or stop

Do not over-chase. A weak-fit lead should not consume founder time indefinitely.

## Red Flags

- user does not log in after two clear attempts
- user only asks for unrelated features
- user wants payroll, GPS, QR, SMS, or ERP scope before proving core value
- user expects compliance approval
- user uploads sensitive files without understanding review boundaries
- user cannot identify a real dispatch or allocation problem
- feedback is curiosity-only and not operational

## Paid-Pilot Decision Criteria

A lead becomes paid-pilot eligible when several of these are true:

- logged in more than once
- completed at least one workflow
- identified a real business problem
- asked for adaptation or further setup
- would involve another decision-maker
- sees value beyond curiosity
- accepts decision-support boundaries
- understands that setup quality affects output quality

## Do-Not-Build List

Do not build during this proof period:

- RBAC expansion
- payroll
- timesheet approval
- GPS verification
- QR sign-in
- SMS automation
- invoicing
- accounting integration
- autonomous dispatch
- compliance approval workflows
- AI extraction into live records
- client-facing portals
- broad ERP features

Build evidence first. Automation can wait.
